# YOLO + 行为树 + UI 自动化系统设计方案

## 1. 背景与目标
本项目要构建一个本地自动化系统：使用 YOLO 识别页面目标，使用优先级规则与行为树进行决策，再通过鼠标键盘执行自动化操作。系统需要同时面向两类长期场景：

- 浏览器 / 桌面软件页面自动化
- 游戏窗口自动化

但为了降低首版复杂度，当前确认的落地顺序是：

1. 先在 Windows 环境下完成单场景 MVP。
2. 首个真实场景选择“浏览器 / 桌面软件按钮点击流程”。
3. 先打通固定流程，再逐步升级到多目标优先级决策。
4. 配置结构和模块接口从第一版开始按多场景扩展来设计。

## 2. 范围边界
### 2.1 第一版必须解决的问题
- 绑定目标窗口并持续截图
- 使用 YOLO 识别页面中的关键目标
- 将检测结果转换成页面状态
- 在同一帧识别到多个目标时，根据优先级规则选择当前处理对象
- 使用简化行为树控制执行流程
- 通过鼠标键盘完成点击、输入等基础动作
- 通过桌面 UI 展示当前画面、检测结果、决策结果和运行日志

### 2.2 第一版暂不追求的能力
- 完整跨平台支持
- 复杂可视化行为树编辑器
- 自学习策略与在线训练
- 多模型融合大系统
- 双进程或分布式架构
- 大量场景并行适配

## 3. 总体架构
采用“单体 MVP + 模块化分层”的方案：先用一个 Python 进程把最小闭环跑通，但内部必须按职责清晰拆分，避免后续扩展困难。

整体主链路如下：

`截图采集 → 目标识别 → 状态建模 → 优先级决策 → 行为树调度 → 鼠标键盘执行 → 结果反馈与日志`

建议的核心模块如下：

1. `capture`
   - 负责窗口匹配、窗口绑定、截图采集、区域裁剪。
   - 第一版以 Windows 为主，接口预留跨平台扩展点。

2. `detector`
   - 负责 YOLO 推理与检测结果标准化。
   - 后续可扩展 OCR、模板匹配等其他识别器。

3. `state`
   - 把原始检测结果转换成“页面状态”。
   - 例如：当前页面阶段、是否存在阻塞弹窗、最近动作、连续失败次数等。

4. `priority`
   - 负责候选目标过滤、打分、排序与选中。
   - 解决“同时识别到多个目标时优先处理谁”的问题。

5. `behavior_tree`
   - 负责流程级调度。
   - 解决“当前处于什么阶段、下一步该不该执行、失败后怎么回退”的问题。

6. `executor`
   - 负责鼠标键盘动作执行。
   - 包括点击、双击、输入、快捷键、等待、窗口聚焦等动作。

7. `ui`
   - 负责运行控制、画面预览、参数调整、日志展示与调试开关。

8. `strategy_config`
   - 负责场景配置、目标定义、优先级规则与行为树配置加载。
   - 第一版优先使用 YAML 配置。

## 4. 关键设计原则
- **先打通单场景，再扩多场景。**
- **检测、决策、执行彻底分层。**
- **策略优先配置化，不把规则散落在业务代码里。**
- **先可运行、再可调优、再可扩展。**
- **第一版从一开始就保留可观测性与安全保护。**

## 5. 核心数据模型
### 5.1 场景 `scene`
场景代表一整套自动化任务。建议字段包括：

- `id`：场景唯一标识
- `name`：场景名称
- `window_match`：窗口匹配规则（标题、进程名等）
- `capture_config`：截图频率、区域、缩放参数
- `entry_conditions`：进入场景的条件
- `exit_conditions`：退出场景的条件
- `priority_rules`：优先级打分规则
- `behavior_tree`：流程定义

### 5.2 检测目标 `target`
每个被识别的候选目标应包含统一结构：

- `class_name`
- `confidence`
- `bbox`
- `center`
- `tags`
- `timestamp`

策略侧还应为每类目标配置：

- `min_confidence`
- `roi`
- `cooldown`
- `weight`
- `label`

### 5.3 页面状态 `state`
`state` 是系统理解到的当前页面状态，而不是 YOLO 的原始检测结果。建议至少包含：

- 当前可见目标列表
- 当前页面阶段
- 阻塞类目标是否存在
- 最近一次动作及时间
- 最近一次动作结果
- 连续失败次数
- 当前窗口是否匹配

### 5.4 动作 `action`
建议把执行层统一抽象为标准动作：

- `click`
- `double_click`
- `move`
- `input_text`
- `hotkey`
- `wait`
- `scroll`
- `focus_window`

动作统一携带参数，例如目标坐标、偏移量、持续时间、重试次数、失败回退策略等。

## 6. 多目标优先级决策机制
多目标优先级决策是本项目的关键能力之一。第一版建议采用“过滤 → 打分 → 排序 → 选中”的机制。

### 6.1 过滤
先排除不应处理的目标：
- 置信度不足
- 不在 ROI 中
- 当前场景阶段不允许处理
- 仍处于 cooldown
- 最近多次失败，被临时禁用

### 6.2 打分
对剩余候选目标计算总分：

`总分 = 类别权重 + 场景加权 + 位置加权 + 时序加权 + 自定义条件加权`

例如：
- 关闭阻塞弹窗：+100
- 主流程按钮：+80
- 次要提示按钮：+30
- 最近 2 秒刚点过：-50
- 位于主操作区：+20

### 6.3 排序与选中
按分数从高到低排序。分数接近时再参考：
- 置信度
- 与上次操作位置的距离
- 当前行为树分支偏好

最终把排序第一的目标交给行为树做最后确认。

## 7. 行为树与优先级系统的职责划分
这两个系统不能互相替代：

- **优先级系统**负责“多个目标里先处理谁”
- **行为树**负责“当前流程该不该做这个动作，以及失败后怎么办”

因此推荐的控制顺序是：

1. YOLO 产出多个检测目标
2. `state` 抽象出当前页面状态
3. `priority` 计算当前候选目标分数
4. `behavior_tree` 结合状态与候选目标决定是否执行
5. `executor` 真正发出动作

## 8. 运行循环设计
第一版采用稳定的轮询主循环：

1. 定位目标窗口
2. 截图采集
3. YOLO 推理
4. 生成检测结果列表
5. 构建页面状态
6. 计算候选目标优先级
7. 交给行为树决定下一动作
8. 由执行器发出动作
9. 记录日志与结果
10. 进入下一轮

循环要求：
- 周期短、可暂停、可停止
- 每轮都有可追踪日志
- 能支持 dry-run 模式
- 在页面无变化时具备退避或暂停策略

## 9. 死循环与误操作防护
自动化系统必须从第一版开始具备安全保护：

- `cooldown`：同一目标短时间内禁止重复操作
- `action_history`：记录最近动作，避免高频重复
- `failure_counter`：连续失败后降级或暂停
- `state_change_check`：动作后检查页面是否变化
- `window_guard`：执行前确认当前窗口仍匹配
- `dry_run`：只识别和决策，不真实操作
- `panic_stop`：随时停止运行

## 10. UI 设计
第一版 UI 的目标不是复杂编辑器，而是“可运行 + 可观察 + 可调参数”。推荐使用 `PySide6`。

建议界面分为 5 个区域：

1. **运行控制区**
   - 选择场景
   - 绑定窗口
   - 启动 / 暂停 / 停止
   - 设置截图频率与推理频率

2. **实时画面区**
   - 显示当前截图
   - 叠加 YOLO 检测框
   - 显示类别名、置信度、优先级分数
   - 高亮当前选中目标

3. **状态面板**
   - 当前场景
   - 当前行为树节点
   - 当前页面状态
   - 最近动作
   - 连续失败次数
   - 当前窗口匹配状态

4. **策略配置区**
   - 最小置信度
   - 类别权重
   - cooldown
   - ROI
   - 动作等待时间
   - 重试次数

5. **日志区**
   - 检测日志
   - 决策日志
   - 执行日志
   - 异常日志

## 11. 日志与调试体系
第一版必须具备四类日志：

1. **检测日志**：检测到的目标、类别、置信度、位置
2. **决策日志**：候选列表、每项分数、最终选中对象
3. **执行日志**：动作类型、坐标或按键、执行时刻、执行结果
4. **异常日志**：窗口丢失、截图失败、推理异常、状态卡死等

调试模式建议保留：
- 单步运行
- 只识别不执行
- 单独执行测试动作
- 关键帧保存
- 动作前高亮目标

## 12. 分阶段实施路线
### 阶段 0：工程骨架
- 项目结构
- 配置加载
- 日志系统
- 模块接口
- 空主循环

### 阶段 1：单场景识别与执行 MVP
- 窗口绑定
- 截图
- YOLO 识别
- 单目标选择
- 点击 / 输入
- UI 基础画面与日志

### 阶段 2：状态建模 + 简化行为树
- 页面状态抽象
- 条件判断
- 基础流程控制
- 阻塞弹窗优先处理

### 阶段 3：多目标优先级决策
- 候选过滤
- 权重打分
- 排序与选中
- cooldown
- 历史动作抑制
- 失败降级

### 阶段 4：策略配置化与 UI 增强
- 场景配置
- 权重规则配置
- 行为树配置
- 参数编辑
- 调试模式增强

### 阶段 5：识别增强与多场景扩展
- OCR 融合
- 模板匹配融合
- 多场景切换
- 游戏窗口扩展
- 更强的回放与复盘工具

## 13. MVP 结论
当前最合适的执行路径是：

**先用 Python 做一个 Windows 下的单场景可运行 MVP，架构上按多场景、可配置、多目标优先级决策系统来设计，再逐阶段补齐行为树、策略配置和调试能力。**

这个路径兼顾了：
- 尽快落地
- 后续可扩展
- 复杂度可控
- 符合当前项目空目录、从 0 搭建的现实情况
